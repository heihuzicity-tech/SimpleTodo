# 船长待办开发指南

这些规则用于帮助后续 AI 或开发者在当前代码库中保持一致，不把历史实现或未接入能力误写成当前事实。

## 架构约束

- 当前产品是 Tauri 桌面应用，持久化主路径是前端 API 封装调用 Rust command，再写入 SQLite。
- 新增项目、列、卡片或设置类数据读写时，应先扩展 `src/lib/api` 和 `src-tauri/src/commands`，不要让业务组件直接处理持久化细节。
- 组件应通过 Hooks 获取业务操作，避免在组件中直接调用 Tauri `invoke`。
- 拖拽主路径基于 `@dnd-kit`，维护拖拽时要同时验证同列移动、跨列移动和空列投放。

## 前端约定

- `App.tsx` 是当前看板主流程入口，文档中描述“已支持功能”时应优先确认该能力是否接入这里。
- `useProjectStore` 负责项目状态，`useKanbanStore` 负责看板状态和卡片/列操作。
- `src/components/ui` 是基础组件层，业务组件优先组合现有基础组件。
- 卡片详情编辑应集中在 `CardDetailsDialog`，卡片本体保持轻量和可扫读。

## 文档约定

- README 面向使用者，优先描述当前可运行、可验证的功能。
- `docs/项目说明.md` 面向维护者，集中记录架构、数据流、关键目录和维护约定。
- 未来规划、实验组件和未接入主流程的能力必须明确标注，不能写成当前产品能力。

## 质量检查

- 文档更新后，搜索历史网页版描述和未接入能力，确认它们没有被当作当前事实保留。
- 保持文档简洁，优先维护 README 和 `docs/项目说明.md`。
- 纯文档更新不需要运行应用构建；如果修改代码或配置，再按变更范围补充构建或测试。
